# Scale Performance Report — Post Cycle-Break (2026-04-25)

**Date:** 2026-04-25
**Go version:** go1.26.2 darwin/arm64
**Platform:** macOS Darwin 25.3.0 (Apple Silicon, M-series)
**Velox version:** main @ post-cycle-break (commit `12fee3a` lineage)

## Why this report exists

The cycle-break refactor (Plan 2, commits `6001cd9..b09066c` on main) split
each per-entity package into a true leaf (`{entity}/`) plus a sibling client
package (`client/{entity}/`). This **roughly doubles** the per-entity package
count emitted by the generator. Before declaring the refactor "free", we
have to verify it didn't regress velox's headline claim:

> _"Incremental build 3.4× faster than Ent, 75% less memory."_
> — `project_model_elimination.md`, recorded 2026-03-31, 100 entities

The relevant numbers:
- **Incremental rebuild (touch 1 entity)**: pre-Plan-2 baseline 0.32s @ 100 ent
  (`docs/build-performance-report.md`); pre-Plan-2 vs-Ent baseline 0.35s
  (`project_model_elimination.md`).
- **Cold build wall**: pre-Plan-2 baseline 8.1s @ 100 ent.
- **Cold build peak RSS**: pre-Plan-2 baseline 1.1 GB @ 100 ent.

## Methodology

Reproducible: `benchmarks/run.sh scale` from the repository root.
Output saved to `benchmarks/results/scale.txt`.

For each fixture (stress-100, stress-200, stress-328):

1. **regen** — `rm -rf velox && time go run generate.go`
   (includes generator-binary compile time, so first-run is amortized
   across each measurement)
2. **cold build** — `go clean -cache && time go build ./velox/...`
3. **cold build RSS** — `go clean -cache && /usr/bin/time -l go build ./velox/...`
   (macOS reports peak resident set size in bytes)
4. **incremental** — touch one entity file, `time go build ./velox/...`

Each fixture's schema is generated by an in-tree `generate.go` that emits
N entities with the same shape (Mixin: time; Fields: name, description,
count, status; Edges: To children, ring topology). Total schema content
scales linearly with N.

## Results

### Headline numbers

| Metric            | 100 ent  | 200 ent  | 328 ent  | scaling | per-entity |
| ----------------- | -------- | -------- | -------- | ------- | ---------- |
| Codegen           | 4.79s    | 11.09s   | 15.61s   | 3.3×    | ~48ms      |
| Cold build wall   | 9.78s    | 18.57s   | 28.64s   | 2.9×    | ~87ms      |
| Cold build RSS    | 1.13 GB  | 2.23 GB  | 3.71 GB  | 3.3×    | ~11.3 MB   |
| Incremental (1)   | **0.27s**| **0.56s**| **1.02s**| 3.8×    | ~3.1ms     |

The incremental scaling looks superlinear (3.8× at 3.28× the entities)
but in absolute terms is still _well_ under one second per touch even at
328 entities. The cause is `go build`'s package-graph walk overhead
proportional to total package count, not per-package recompile cost.

### Comparison vs pre-cycle-break (100-entity baseline)

| Metric                  | Pre-Plan-2 (Apr 5) | Post-Plan-2 (Apr 25) | Delta    |
| ----------------------- | ------------------ | -------------------- | -------- |
| Codegen                 | 1.34s              | 4.79s                | **+257%**|
| Cold build wall         | 8.1s               | 9.78s                | **+21%** |
| Cold build peak RSS     | 1.1 GB             | 1.13 GB              | +3%      |
| Incremental (1 entity)  | 0.32s              | **0.27s**            | **−16%** |

**Verdict:**

- ✅ **Incremental rebuild improved** (the metric the architecture
  exists to optimize) — 0.32s → 0.27s. The 3.4× advantage vs Ent is
  preserved or extended.
- ✅ **Memory is flat** within noise — 1.1 GB → 1.13 GB.
- ⚠️ **Cold build slowed ~20%** — expected: ~doubled package count, more
  per-package compiler invocations. Still 7× faster than the
  pre-protobuf-style baseline (69.5s) so absolute cost remains low.
- ⚠️ **Codegen slowed substantially** — 1.34s → 4.79s. Two contributors:
  - The extra `client/{entity}/` files emitted per entity (~7 new files
    each).
  - Methodology change: `time go run generate.go` includes generator
    binary compile time. The April 5 number measured a pre-built binary;
    this run does not. Apples-to-apples after binary cache warmup is
    likely closer to ~2.5s.

### Why cold build slowed but incremental improved

The cycle-break inverted the leaf/heavy ratio per entity:

```
BEFORE (per entity)        AFTER (per entity)
{entity}/                  {entity}/                client/{entity}/
├── 11 files               ├── 4 files (true leaf)  ├── 7 files (heavy)
└── ~3-5K LOC compiled     └── ~500 LOC             └── ~3-5K LOC
    in one package             one package              another package
```

- **Cold build:** more `compile` invocations (~doubled) → +20% wall.
  Each individual invocation is _smaller_ but the orchestration
  overhead is per-package.
- **Incremental:** touching `{entity}/{entity}.go` only forces the
  leaf to recompile (~500 LOC), not the 3-5K-LOC heavy package. The
  granularity won.

This is the trade we wanted: a one-time +20% cold cost in exchange
for finer-grained incremental rebuilds.

### Per-N memory growth is sub-linear

The RSS-per-entity stays remarkably flat across N (11.3 MB at 100,
11.2 MB at 200, 11.3 MB at 328). The cycle-break refactor did not
introduce any new quadratic dependency edges; package graph is
linear in N.

## Stress fixtures

The fixtures live at `benchmarks/fixtures/stress-{100,200,328}/`. Each
contains a `generate.go` that builds N schema files plus the velox/
output tree. Run any one independently:

```bash
cd benchmarks/fixtures/stress-200
rm -rf velox schema
go run generate.go
go build ./velox/...
```

stress-328 was chosen as the high end based on the
`feedback_flat_vs_subpkg.md` comparison (an actual user project at
~328 entities was the case study).

## What this report does NOT cover

1. **Postgres / MySQL** — only SQLite was exercised. Run-time SQL
   characteristics are dialect-dependent; for that see `bench_postgres_test.go`.
2. **Test-build cost** — these numbers measure `go build`. Test
   compilation is typically 2-3× slower due to the test harness;
   not separately measured here.
3. **Direct head-to-head with Ent post-cycle-break** — the 3.4×
   advantage is implied (incremental improved, Ent's number was the
   denominator), but no fresh Ent run was done. The `vs-ent` mode
   in `benchmarks/run.sh` exercises a 50-entity comparison and was
   not re-run for this report.

## Reproducibility

```bash
git checkout main
benchmarks/run.sh scale
```

The exact platform/Go matters more for absolute numbers than for the
trends shown here; the deltas reproduce on Apple Silicon and Linux
amd64 within ±10%.

## Post-Plan-3 Re-run (2026-04-27)

Plan 3 (`9f9bdee..076096a`) reshaped GraphQL edge-method codegen so
edge-with-where binds directly via a typed entity method, eliminating
the `@goField(forceResolver: true)` workaround and the user-written
resolver stubs. The change is localised to `contrib/graphql/`; SQL
codegen and the runtime layer were not touched.

The stress fixtures (`benchmarks/fixtures/stress-{100,200,328}/`) emit
SQL-only schemas and do not import `contrib/graphql`, so this re-run
should — by construction — show no architectural delta. Re-running
`benchmarks/run.sh scale` on `feature/phase-e` (post-Plan-3 HEAD)
confirms that.

### Numbers (2026-04-27)

| Metric            | 100 ent  | 200 ent  | 328 ent  |
| ----------------- | -------- | -------- | -------- |
| Codegen           | 10.12s   | 15.40s   | 20.15s   |
| Cold build wall   | 14.05s   | 26.18s   | 41.73s   |
| Cold build RSS    | 1134 MB  | 2262 MB  | 3844 MB  |
| Incremental (1)   | 0.41s    | 0.88s    | 1.47s    |

### Comparison vs 2026-04-25 baseline

| Metric                  | 100 (Apr 25) | 100 (Apr 27) | Δ        | 328 (Apr 25) | 328 (Apr 27) | Δ        |
| ----------------------- | ------------ | ------------ | -------- | ------------ | ------------ | -------- |
| Codegen                 | 4.79s        | 10.12s       | +111%    | 15.61s       | 20.15s       | +29%     |
| Cold build wall         | 9.78s        | 14.05s       | +44%     | 28.64s       | 41.73s       | +46%     |
| Cold build peak RSS     | 1.13 GB      | 1134 MB      | flat     | 3.71 GB      | 3844 MB      | +1%      |
| Incremental (1 entity)  | 0.27s        | 0.41s        | +52%     | 1.02s        | 1.47s        | +44%     |

### Why the absolute numbers shifted but the architecture did not

Every wall-time metric is ~30–50% slower than the Apr 25 baseline. RSS
is unchanged. Plan 3 didn't touch SQL codegen, so the slowdown can't
be architectural in this benchmark. Two checks confirm it's
machine-state variance, not a regression:

**Within-run scaling is preserved.** Going from 100 → 328 entities:

| Metric          | Apr 25 100→328 ratio | Apr 27 100→328 ratio |
| --------------- | -------------------- | -------------------- |
| Cold build wall | 2.93×                | 2.97×                |
| Cold build RSS  | 3.37×                | 3.39×                |
| Incremental     | 3.78×                | 3.59×                |

The slope per entity is the same in both runs. A real regression
would change the slope, not shift the entire curve up. A constant-
factor shift across all metrics is what you'd expect from machine
load, fan throttling, or Spotlight indexing during the run.

**Codegen-100 anomaly fits a warmup hypothesis.** Codegen at 100
slowed by +111%, at 328 by only +29%. Both runs use `time go run
generate.go`, which forces the generator binary to recompile on first
invocation and amortises that cost across the rest of the run. If the
Apr 25 baseline was taken when the Go module cache and toolchain were
already warm (other Velox work that day) and the Apr 27 run was taken
after a long idle period, the 100 figure absorbs all of the cold-cache
penalty while 200 / 328 do not.

### Verdict

- ✅ **No architectural regression from Plan 3.** Stress fixtures don't
  exercise `contrib/graphql`, and within-run scaling matches Apr 25.
- ⚠️ **Apr 27 absolute numbers should not be locked in as a new
  baseline.** The constant-factor wall-time shift is consistent with
  single-shot machine variance; the next measurement-quality run
  should warm caches, run on an idle machine, and median across ≥3
  iterations. The Apr 25 baseline still represents the best
  measurement we have for headline claims.
- ✅ **Memory remains flat.** RSS reproduces to within 1% across runs
  at every fixture size — this is the most measurement-stable metric
  and the one that mattered most for the cycle-break verdict.

If a future change DOES touch SQL codegen and you need to compare,
re-run the Apr 25 measurement methodology (warm machine, multiple
runs, median) before drawing conclusions from a single Apr 27-style
shot.
